# weather/

日本旅游天气预测，30 天视野，支持**城市**和**山地/具体地点**两种精度。不需要
cookie、不需要浏览器，纯调 [Open-Meteo](https://open-meteo.com/) 公开 API。

## 为什么要分"城市"和"具体地点"两级

城市中心点的预报不能代表几十公里外、海拔差出上千米的山区实际情况——这正是这次
改造的起因：立山黑部阿尔卑斯路线上，美女平（977m）、弥陀之原（1930m）、室堂
（2450m）三个站点之间海拔差超过 1400 米，天气可以完全不同；户隐神社离长野市区
（海拔约 370m）有约 20km 山路、海拔约 1200m，市区天气同样不能直接套用。

`locations.py` 里的 `MOUNTAIN_POINTS` 就是给这类地点用精确坐标 + 显式海拔查询，而
不是查最近城市了事。

## 调研结论：换了什么、为什么没换成 tenki.jp

调研过程见下，先说结论：

- **保留 Open-Meteo 作为程序化数据源，但把模型从隐式的多模型综合（`best_match`）
  换成显式的 `jma_seamless`**——日本气象厅（JMA）自有数值模式专门按日本地形调校，
  对日本境内的预报通常比通用全球模型更准，这是日本几乎所有商业天气网站（tenki.jp、
  Yahoo!天气等）预报数据的共同源头。16 天内：`jma_seamless`（约覆盖 11 天，逐日
  取用）→ 缺口用 `best_match` 多模型综合补齐到 16 天。16 天外没有任何机构能给出
  可信的逐日确定性预报（这是气象学可预报性上限，不是数据源选择问题），保留原有的
  最近 3 整年同期历史均值兜底（年份改成动态计算，不再写死，避免过几年自动过期）。

- **新增 `elevation` 显式海拔参数，这是"精确到地点"真正的关键，比切换模型本身更
  重要**——Open-Meteo 默认按 90m 数字高程模型（DEM）自动判定查询点海拔，但山区
  地形陡峭时自动判定可能大幅偏离实际。实测美女平（真实海拔 977m）：不传
  `elevation` 参数时，Open-Meteo 自动选用附近一个 **2225m** 的网格点，算出当天
  最高气温只有 **14.3°C**；显式传入 `elevation=977` 后修正为 **21.0°C**——足足差了
  6.7°C，如果不做这个修正，山脚站点会被当成山顶来预报。`MOUNTAIN_POINTS` 里的
  每个点都手动标了真实海拔，就是为了拿到这个修正。

- **没有接入 tenki.jp（或其他日本消费级天气站）的山地/景点页面做爬虫补充**：
  实测抓取 tenki.jp 美女平"おでかけスポット天気"页面（`/leisure/4/19/126/46933/
  10days.html`）后发现，页面 HTML 注释里写着数据来源是
  `name:立山町 jiscode:16323`——也就是说这个看起来"专属美女平"的页面，实际给的
  就是立山町这一整个行政区划的通用预报，并不比城市级预报更精细，不会带来精度
  增量，接入它反而可能让人误以为这是"专属该地点"的预报。日本消费级天气站的
  "景点天气"普遍是这个模式（复用市町村级 JMA 预报换个壳），所以坐标+海拔传给
  Open-Meteo 做数值降尺度，实际上已经比爬这些站点更精细。
  真正比通用数值预报更懂地形效应的是
  [ヤマテン（山の天気予报）](https://i.yamatenki.co.jp/)——由专业山岳气象预报员
  手工订正、登山者公认精度最高，立山所在的"北アルプス北部"在其"特别预报"覆盖
  范围内。但它是付费订阅制、无公开 API、需要登录，不适合也不应该自动化抓取。
  **建议**：立山穿越、雪之大谷徒步这类关键行程日，出发前手动上 Yamaten 或
  [tenki.jp 山の天気](https://tenki.jp/mountain/) 做人工二次确认，本工具给的是
  可编程、可批量查、可提前一个月看趋势的基线参考，不是替代专业山岳预报。

## 地点坐标是怎么定的

`MOUNTAIN_POINTS` 里每个点的坐标/海拔来源和置信度不同，标在 `locations.py` 对应
行的注释里：室堂（2450m）、美女平（977m）是路线官方公开的常识数据，置信度高；
户隐神社（中社）坐标经检索核实为 `36.742472, 138.085135`，海拔约 1200m；弥陀之原、
雪之大谷精度弱于前两者（雪之大谷视同室堂坐标）。新增地点时优先找官方/维基百科
给出的车站或设施海拔，不要凭感觉估。

## 文件

- **`locations.py`** — `CITIES`（城市，粗粒度，海拔交给 Open-Meteo 自动判定）+
  `MOUNTAIN_POINTS`（具体地点，显式海拔）坐标库，以及 `resolve_city()` /
  `resolve_point()` 查找函数。
- **`open_meteo.py`** — 三级 fallback 的实际请求逻辑（`fetch_jma_forecast` /
  `fetch_best_match_forecast` / `fetch_historical_range` / `compute_historical_avg`）+
  WMO 天气代码表。
- **`forecast.py`** — 编排层：解析命令行 → 按 `--point` > 手动坐标 > `--city`
  优先级算出坐标/海拔 → 合并三级数据 → 输出 md/json。`main()` 是 CLI 入口。

## 调用方式

```bash
# 城市（原有用法不变）
python3 cli/weather_forecast.py --city kobe --days 14 --format md

# 内置山地/具体地点
python3 cli/weather_forecast.py --point murodo --days 10 --format md
python3 cli/weather_forecast.py --point bijodaira --days 10 --format md
python3 cli/weather_forecast.py --point togakushi --days 10 --format md

# 列出所有内置地点
python3 cli/weather_forecast.py --list-points

# 任意坐标 + 手动海拔（不在内置列表里的地点）
python3 cli/weather_forecast.py --latitude 36.57 --longitude 137.57 --elevation 977 --days 10
```

`--days` 默认 30；超过 16 天的部分自动退化成历史同期均值，输出的"数据来源"列会
标出每一天到底是 `🎌 JMA精细预报` / `🌐 多模型综合预报` / `📜 历史同期均值`
哪一档，不是所有天数据可信度都一样，看这一列。

## 已知限制

- 30 天视野里，只有前 16 天左右是真正的逐日预报，之后是历史均值——这是气象学
  本身的可预报性上限，不是本工具的短板，任何声称能精确预报一个月后某一天天气的
  工具都不可信。
- `jma_seamless` 只覆盖日本区域，本工具的坐标库/默认用法也只面向日本旅游场景。
- 山地天气本身多变，本工具的数值预报仅供行前趋势参考；真正决定登山/徒步当天
  去留，请按上面"调研结论"里说的人工核对 Yamaten 等专业山岳气象来源。
