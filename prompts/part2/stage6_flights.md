# Stage 6: 路线分析 + 机票价格矩阵

**输入**: 出发地/出发机场 + 目的地区域 + 日期窗口 + 用户偏好（中转限制等）（参见 `prompts/globals.md`）

**任务**: 分析从出发地到目的地区域的实用路线方案 → 按方案查机票 → 结合地面接驳给出落地总成本对比与结论。

---

## Step 1: 路线分析（先做，不查价）

根据 globals.md 的出发地和目的地区域，推导实用的航空路线方案。

**执行步骤**:
1. 用 WebSearch 搜索目的地区域附近有哪些可用的国际/国内机场
2. 对每个候选门户机场，搜索从该机场到目的地核心城市的地面交通方式（铁路/巴士）、大约耗时和**单程票价概算**
3. 排除地面交通单程超过 4-5 小时的机场（除非出发地直飞航线极少，无其他选择）
4. 根据目的地区域的地理走向，设计路线方案：
   - **同机场往返**: 每个实用门户机场各一个方案
   - **Open-jaw（不同机场进出）**: 只设计地理上合理的组合——进出机场分布在目的地两侧、形成穿越路线、不走回头路的才算合理

**路线分析表**（中间产物，写入 locked 文件；耗时/票价需附出处链接）:

| 门户机场 | 地面走廊 | 到目的地核心城市耗时 | 地面单程概算票价 | 实用性 |
|---|---|---|---|---|
| （从搜索结果填写） | | | （日元，附出处） | 主力 / 备选 / 仅价差极大时考虑 |

| 路线方案 | 进入机场 | 地面路线概要 | 离开机场 | 方案特点 |
|---|---|---|---|---|
| （从地理分析推导） | | | | 如"南→北穿越，不折返" |

---

## Step 2: 按路线方案查机票

**工具**: `cli/ctrip_flights.py`，在仓库根目录执行，有头模式（默认即有头，不许加 `--headless`）。

**工具现实（必读，2026-07 实测）**:
- 携程**往返列表页抽取不到数据**（页面要选完去程才出回程价）。所有查价一律按**单程**查，去程+回程两段拼接为方案总价。`--return` 往返模式只可用于附加参考截图，不许作为报价来源。
- 工具**默认只保留直飞**。必须加 `--all-flights`，否则漏掉全部中转航班（低价票常是中转）。
- 机场一律用**具体机场三字码**（can / nrt / hnd / ngo / kix …），不用城市码（tyo / osa），避免城市码 fallback 产生噪音。
- 列表页卡片**不含航班号**（航班号按字段规则另行核实），价格为**含税起价**。

**日期规则**:
- 日期窗口 = 固定行程日期（窗口长度 = 行程天数）→ 只查这一组去程/回程日期
- 日期窗口 > 行程天数（"任选 N 天"型）→ 先选 1-2 个主力航线，对每个合法出发日/回程日批量查"价格 × 日期"矩阵（`--query` 多条即可），收敛出 2-3 组候选日期，再对全部路线方案查价

**命令模板**（一次浏览器会话批量查完全部航段，raw 三件套同名保存到 `output/plan/raw/`）:

```bash
python3 cli/ctrip_flights.py \
  --query can-nrt,2026-12-16 --query nrt-can,2026-12-22 \
  --query can-hnd,2026-12-16 --query hnd-can,2026-12-22 \
  --all-flights --adults 1 --format json --debug \
  --screenshot output/plan/raw/stage6_oneways.png \
  > output/plan/raw/stage6_oneways.json \
  2> output/plan/raw/stage6_oneways.stderr
```

（`--query` 每个航段一条：每个门户机场的去程+回程各一条；`--adults` 按 globals 人数；截图会自动展开为 `stage6_oneways-NN-<from>-<to>-<date>.png`）

**筛选与候选规则**:
- **中转合规过滤**: 按 globals 用户偏好过滤（如"只接受中国境内或日本境内中转"）。凡被排除、但价格低于该航线入选最低价的航班，必须记入排除表，不许静默丢弃
- **每条航线（单向）记 2-3 个候选**，不许只记最低价：
  1. `最低价`: 合规最低价航班
  2. `接驳优先`: 去程 = 到达最早的合规航班（直飞优先）；回程 = 出发最晚的合规航班（直飞优先）
  - 两者是同一班就只记一行，标 `最低价+接驳优先`
- **失败处理**: stderr 出现"验证"（风控拦截）→ 在自己的 Chrome 打开一次携程后重跑；重试一次仍失败才标 `unknown`，不删行

## Step 3: 接驳可行性 + 落地总成本

**去程当日接驳**: 对每个去程候选，用 WebSearch 查关键末班接驳（如 新宿→松本 末班あずさ、东京→长野 末班新干线、机场→市区枢纽耗时），按 **到达时刻 + 90 分钟出关提行李** 推算能否当日到达目的地核心城市，标注 `可当日达(末班XX xx:xx发)` 或 `需中途一晚(东京/名古屋/大阪)`。判断链条要写出来（如 `18:40着 +90min → 20:10 NEX → 新宿21:4x，已过末班あずさ20:00 → 需中途一晚`），末班车信息附出处+查询日期。

**回程最晚离开**: 对每个回程候选，按 **起飞前 2.5 小时到机场 + 地面耗时** 倒推"最晚需离开目的地核心城市的时刻"。

**落地总成本** = 机票两段拼接总价 + 地面往返概算（Step 1 票价，日元按 WebSearch 当期汇率折 CNY，注明汇率与出处）+ 需中途一晚时按 ¥400/晚 计入（标注"概算"）。

## 输出字段规则

**机场名**: 遵循 globals.md 命名规则，中文机场名(IATA)。

**航班号**: 携程列表页不含航班号——这不是没查，是页面没有。规则：用 WebSearch 按"航司 + 航线 + 出发时刻"核实（如"南方航空 广州 羽田 08:50 航班号"），核实到的填入并附出处；核实不了写 `unknown`。不许按航司号段猜。

**行李额**: LCC（春秋、香港快运、乐桃、捷星等）用 WebSearch 查该航司基础票价的默认托运政策，标注如 `不含托运(手提7kg)`，附出处；全服务航司写 `含托运`；查不到写 `unknown`。汇总表概要里带上行李差异，低价 LCC 与全服务航司比价时不许无视行李口径。

**时间**: 出发 = 出发地当地时间，到达 = 目的地当地时间，24h 制，标 (+0) 当日到 / (+1) 次日到。

**直飞/中转**: 直飞写 `直飞`；中转写 `中转(经XX, N次)`（工具原样输出）。行李直挂/换航站楼信息查得到才补充，查不到不占位。

**航站楼**: `出发T→到达T`，如 `T2→T1`（工具可从卡片抓到）。查不到写 `unknown`。

**价格口径**: 一律为携程列表含税起价（"¥xxxx起"），实付以预订页为准。每张表带查询时间。

---

## 禁止

- 不许编造价格、航司、航班号、时间、末班车、汇率——WebSearch 补充的数据（航班号/行李额/末班车/地面票价/汇率）必须附出处链接 + 查询日期
- 每行价格必须能对应 `output/plan/raw/` 下的 JSON/截图；locked 文件开头列出本次全部 raw 文件路径
- 查询失败标 `unknown`，不删除该行
- 不许跳过 Step 1 路线分析直接查价；不许跳过 Step 3 只按机票价排序——汇总表必须含落地总成本列
- 不许用 `--return` 往返模式的抽取结果报价（当前抽不到数据）；不许漏 `--all-flights`
