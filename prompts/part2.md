# Part 2: 决策规划（消费 Part 1 锁定数据）

> 参数参见 `prompts/globals.md`，Part 1 锁定产物在 `output/plan/stage{1-5}_locked.md`。

决策排序用**字典序**（不搞加权黑盒）:
1. 硬约束（航班存在、必去项在窗口内开放、每日行程物理可达）
2. 总成本（机票 + 酒店）
3. 体验（天气命中率、特殊活动、拥挤度）
4. 鲁棒性（每个基地日各池是否有备选）

### Stage 6: 机票价格矩阵

**输入**: 日期窗口 + 出发地 + 候选门户机场
**任务**: 只查机票，不查酒店。对所有候选窗口 × 所有候选门户 × 进出方向，查单程机票价格。

**候选门户不要预设**，让数据裁决。示例: {TYO(HND/NRT), NGO, KIX}。
**必须按单程查、允许开口（open-jaw）**: A 城进 B 城出可能省掉折返。

**工具**: `cli/ctrip_flights.py`（必须通过有头浏览器操作，不许用 headless 模式）

**输出格式**——完整矩阵，每个窗口 × 每个门户组合一行:

| 窗口 | 星期 | 去程(出发地→门户) | 去程价格 | 回程(门户→出发地) | 回程价格 | 往返总价 | 查询时间 |
|---|---|---|---|---|---|---|---|
| 12.16-22 | 三~二 | CAN→NRT | ¥xxxx | NRT→CAN | ¥xxxx | ¥xxxx | 2026-xx-xx |
| 12.16-22 | 三~二 | CAN→NGO | ¥xxxx | NGO→CAN | ¥xxxx | ¥xxxx | 2026-xx-xx |
| 12.16-22 | 三~二 | CAN→NRT(去) NGO→CAN(回) | ¥xxxx | - | ¥xxxx | ¥xxxx | 2026-xx-xx |
| …所有组合 | | | | | | | |

**禁止**: 不许编造价格。每行必须能对应 `output/plan/raw/` 下的原始查询结果文件。查询失败标 `unknown`，不删除该行。所有窗口 × 所有门户组合都要列出，不许只列最便宜的。

完成后将完整结果表格输出，然后将结果写入 `output/plan/stage6_locked.md`。

### Stage 7: 酒店价格矩阵

**输入**: 日期窗口 + 候选基地城市
**任务**: 只查酒店，不查机票。对每个基地城市，按 3 个档次各挑 6 家酒店，查每一天的价格。

**档次定义**:
- **高端**: 温泉旅馆、高档度假酒店、一泊二食等（该城市价格前 20%）
- **中等**: 标准商务酒店、评分较高的连锁（该城市价格中间 40%）
- **平价**: 青旅、胶囊、价格最低的商务酒店（该城市价格后 20%）

**工具**: `cli/ctrip_hotels.py` / `cli/expedia_hotels.py`（必须通过有头浏览器操作，不许用 headless 模式）

**输出格式**——对每个基地城市输出一张表，每个档次 6 家:

**松本市**:

| 档次 | 序号 | 酒店名称 | 12/16价格 | 12/17 | 12/18 | 12/19 | 12/20 | 12/21 | 来源链接 | 查询时间 |
|---|---|---|---|---|---|---|---|---|---|---|
| 高端 | 1 | xxx温泉旅馆 | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | （预订页链接） | 2026-xx-xx |
| 高端 | 2 | … | … | … | … | … | … | … | … | … |
| …高端共6家 | | | | | | | | | | |
| 中等 | 1 | xxx商务酒店 | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | （预订页链接） | 2026-xx-xx |
| …中等共6家 | | | | | | | | | | |
| 平价 | 1 | xxx青旅 | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | ¥xxxx | （预订页链接） | 2026-xx-xx |
| …平价共6家 | | | | | | | | | | |

**长野市**: （同上格式，3 档 × 6 家）
**上諏訪**: （同上格式，3 档 × 6 家）
…每个候选基地城市一张表。

**禁止**: 不许编造价格和酒店名。每行必须能对应原始查询结果。查不到的天标 `unknown`。不许只给中位价——必须列出具体酒店名和逐日价格。不许合并档次或减少酒店数量。

完成后将完整结果表格输出，然后将结果写入 `output/plan/stage7_locked.md`。

### Stage 8: 交通骨架

放在机票酒店之后：先确定价格矩阵，再查交通——交通骨架需要知道候选门户和基地城市，这些在 Stage 6-7 中已经明确。

**输入**: Stage 5 锁定的最终景点列表（`output/plan/stage5_locked.md`） + Stage 6-7 确定的候选门户和基地城市
**任务**: 每个景点从最近的基地城市怎么到达。同时标注有当日运行风险的项目。

先列出干线结构（一次性，示例）:
```
东京(新宿) —あずさ ~2.5h— 上諏訪/茅野 —~20min— 松本 —しなの/篠ノ井線 ~1h— 長野 —新幹線 ~1.5h— 东京
名古屋 —しなの ~2h— 松本
```

再对每个景点填表:

| 序号 | 名称 | 最近基地 | 交通方式 | 大约耗时 | 当日风险 |
|---|---|---|---|---|---|
| 1 | 立石公園 | 上諏訪 | 出租 10min / 步行上坡 40min | 10-40min | 无 |
| 2 | 地獄谷野猿公苑 | 長野 | 巴士 + 雪道步行 | ~75min | 冬季班次少 |
| 3 | 戸隠 | 長野 | 巴士 | ~60min | 冬季可能停运 |

**当日风险枚举**: `无` / `冬季班次减少` / `冬季可能停运` / `缆车/索道天气停运` / `山路积雪封闭` / `事故/灾害临时封闭`

**有当日风险的项目，自动生成官网检查清单**:

| 名称 | 官网 URL | 出发前查什么 | 当天查什么 |
|---|---|---|---|
| 地獄谷野猿公苑 | （真实官网链接，查不到写 unknown） | 冬季巴士时刻表 | 当日道路状况 |
| 戸隠 | （真实官网链接，查不到写 unknown） | 冬季巴士是否运行 | 当日运行情报 |

**禁止**: 不许增删景点。不许查具体时刻表（预订阶段再查）。耗时写大概值即可。
官网 URL 必须是能点开的真实链接，不许用 xxx 占位交差。

完成后将完整结果表格输出，然后将结果写入 `output/plan/stage8_locked.md`。

### Stage 9: 窗口选择

**输入**: Stage 6 机票矩阵 + Stage 7 酒店矩阵 + Stage 3 开放数据 + Stage 4 活动 + 日历（周末/假日）
**任务**: 按字典序排列所有窗口，选出前 2~3 个进入 Stage 10（不是 1 个——留备胎）。

**输出格式**:

| 排名 | 窗口 | 硬约束 | 总成本 | 活动命中 | 周末夜数 | 入选/淘汰原因 |
|---|---|---|---|---|---|---|
| 1 | 12.x-x | ✓ | ¥xxxxx | 点灯✓ | 1 | 成本最低 + 活动命中 |
| 2 | 12.x-x | ✓ | ¥xxxxx | 无 | 1 | 成本次低 |
| 淘汰 | 12.x-x | ✗ | - | - | - | 周一必去馆全闭 |

**禁止**: 不许只输出赢家。所有窗口都要列出并写明入选/淘汰原因。

完成后将完整结果表格输出，然后将结果写入 `output/plan/stage9_locked.md`。

### Stage 10: 路线与池子设计

**输入**: 前 2~3 个窗口 + Part 1 全部锁定数据 + Stage 8 交通骨架
**任务**: 对前 2~3 个窗口都跑轻量可行性验算，然后锁定主窗口 + 备选窗口。

**步骤**:
1. 确定基地城市（≤ 2 个）和方向（由进出门户决定）
2. 对每个窗口跑**开放日矩阵**（Stage 3 数据 × 具体日期）:

| 景点 | Day1(周x) | Day2 | Day3 | Day4 | Day5 | Day6 |
|---|---|---|---|---|---|---|
| 松本城 | ○ | ○ | ○ | ○ | ○ | ○ |
| xxx美术馆 | ○ | ✗(周一休) | ○ | ○ | ○ | ○ |

3. 检查每日户外容量（冬季日落 ~16:30 → 有效户外 9:00-16:30 ≈ 7.5h → 扣通勤后**每天最多 2 个户外大项**）
4. 将景点分配到池子:
   - **锚点**（已订餐厅、定日活动）→ 钉死日期
   - **晴天池** → 需要能见度/好天气的
   - **雪天池** → 下雪反而更好的
   - **雨天/室内池** → 恶劣天气备选
   - **夜间池** → 日落后活动（点灯、居酒屋）
5. **鲁棒性检查**: 每个基地日，晴天池和雨天池各至少 1 个可达备选。做不到 → 调整基地夜数或回退 Stage 9。

**输出**:
- `primary_window` + `backup_window` + 淘汰窗口及原因
- 基地城市 × 夜数分配
- 开放日矩阵
- 每日容量表
- 四个池子清单
- 需人工核实的班次/道路清单（从 Stage 8 的当日风险项生成）

**禁止**: 不许把景点钉死到具体某天（锚点除外）。池子是策略，不是日程表。

完成后将完整结果输出，然后将结果写入 `output/plan/stage10_locked.md`。

### Stage 11: 预订策略

**输入**: Stage 10 锁定方案
**任务**: 按风险排序（不是时间排序）列出预订动作。

| 优先级 | 对象 | 动作 | 原因 | 取消截止 |
|---|---|---|---|---|
| 1 | 机票 | 立即订 | 供给少，临近必涨 | 按航司政策 |
| 2 | 温泉宿 | 订可退档 | 周末爆满，免费取消 = 免费期权 | xxxx-xx-xx |
| 3 | 餐厅 | 跑 food_discovery 后订 | 订上即成锚点 | - |
| 4 | 商务酒店 | 等待 | 供给充足，留调整余地 | - |

**滚动复查节点**:

| 时点 | 动作 |
|---|---|
| T-16 | 首次逐日天气预报（多模型层），初步池子派发 |
| T-11 | JMA 11 天预报生效，正式派发草案 |
| T-7 | 酒店免费取消截止前最后确认基地分配 |

完成后将完整结果输出，然后将结果写入 `output/plan/stage11_locked.md`。

### Stage 12: 每日执行卡片

**输入**: 当日天气预报 + 池子 + 官网检查结果
**任务**: 每天早上 10 分钟，按天气从池子派发当日菜单。

**卡片格式**:
```
日期: YYYY-MM-DD
天气: 晴/雪/雨 → 采用池: 晴天池

主菜单:
  09:00 出发 → xxx（耗时 xx min）
  13:00 → xxx
  15:30 → 立石公園（16:00 前到位看日落）

备选（主菜单某项失败时）:
  xxx → 替换为 xxx

官网检查:
  [✓] xxx 缆车正常运行（08:30 查）
  [✗] xxx 临时休馆 → 已用备选替换

今天不去: xxx（原因: 能见度差）
```

**原则**: 坏天气毁掉的是某个具体景点，不是那一天——池子保证每天都有备选。

## 全流程一句话总结

> 能确定的写结论，不能确定的写 `unknown` + 怎么查 + 什么时候查，需要人拍板的给出排好序的候选。每一步只做一件事，做完锁住再走下一步。
